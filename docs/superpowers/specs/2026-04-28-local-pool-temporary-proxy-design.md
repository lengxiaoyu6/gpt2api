# 本地账号池临时代理分配设计

## 目标

将本地账号池的代理使用方式调整为“按请求临时租用代理”，覆盖以下两类运行态请求：

1. `chat`
2. `image`

每次请求开始时从代理池挑选一条当前空闲代理，按 `health_score DESC`、`last_used_at ASC`、`id ASC` 排序尝试租用；本次请求结束后释放租约。当前没有可租用的健康代理时，当前请求回退为直连。

## 范围

本次修改仅改变本地账号池在请求执行阶段的代理分配方式，涉及以下部分：

1. 调度器在 `Dispatch(ctx, modelType)` 中按模型类型决定是否申请临时代理。
2. Chat 与图片执行链路在代理层失败时更新代理健康分。
3. 代理池增加 `last_used_at`，用于临时租用排序。
4. 管理后台移除账号级手动绑定入口。

以下能力继续保持现状：

1. `account_proxy_bindings` 继续存在。
2. 导入账号时 `default_proxy_id` 继续可选。
3. 后台刷新、额度探测、图片代理下载继续使用持久绑定代理。

## 现状与根因

当前账号调度器在 `tryLock` 中读取 `account_proxy_bindings`，将绑定代理直接写入 `lease.ProxyURL`。这一模型满足了“单账号固定出口”，但带来两个问题：

1. 请求高并发时，空闲代理无法被其他账号复用，代理利用率偏低。
2. 代理池已有 `health_score` 与探测结果，但运行态请求并未参与临时调度，也未把代理层故障反馈到健康分。

因此需要把“请求期占用代理”和“后台任务使用绑定代理”拆开处理。

## 方案

### 调度器

`Dispatch(ctx, modelType)` 继续保留原有入口。调度器先完成账号锁、鉴权解密、`device_id` 与 `session_id` 补齐，然后按 `modelType` 分支：

1. `chat` 与 `image`：尝试申请临时代理租约。
2. 其他类型：继续读取 `account_proxy_bindings`。

临时代理选择规则：

1. 只在 `enabled = 1` 且 `health_score > 0` 的代理中挑选。
2. 候选顺序按 `health_score DESC`、`last_used_at ASC`、`id ASC`。
3. 对候选逐条尝试 Redis 锁 `proxy:lease:{id}`。
4. 首条抢锁成功的代理写入 `lease.ProxyURL` 与 `lease.ProxyID`。
5. 全部候选因被占用或为空时，当前请求改为直连，`lease.ProxyURL` 为空。

这里的“回退为直连”仅适用于本地账号池的 `chat` 与 `image` 请求，不影响后台刷新、额度探测与图片代理下载。

### 临时代理租约

新增独立的代理分配器，职责固定为：

1. 查询候选代理。
2. 尝试 Redis 锁。
3. 组装代理 URL。
4. 在分配成功时写入 `proxies.last_used_at`。
5. 在请求结束时释放代理锁。
6. 在代理层失败时按规则扣减健康分。

调度器返回的 `Lease` 继续承载账号信息，同时附带临时代理租约释放逻辑。`Lease.Release` 的顺序调整为：

1. 更新账号 `last_used_at` 与当日使用计数。
2. 释放代理锁。
3. 释放账号锁。

进程异常退出时，账号锁与代理锁都由 Redis TTL 兜底回收。

### 健康分更新

代理健康分仅在“代理层失败”时扣减，业务失败不扣减。

扣减规则：

1. 每次请求最多扣减一次。
2. 建议单次扣减 `20` 分。
3. 最低扣到 `0`。
4. `last_error` 记录简短失败摘要。

归类为业务失败、因此保持代理健康分不变的情况包括：

1. 上游 `401`、`403`
2. 上游 `429`
3. 模型拒绝
4. 免费账号静默拒绝
5. 其他已成功到达上游、但由上游业务逻辑返回的失败

归类为代理层失败的情况包括：

1. 代理握手失败
2. 连接代理失败
3. 代理 URL 非法
4. 代理 TLS 协商失败
5. 其他明确发生在代理建立阶段的错误

## 数据结构

`proxies` 表新增字段：

1. `last_used_at DATETIME NULL`

用途如下：

1. 与 `health_score` 共同决定临时租用顺序。
2. 只在临时代理分配成功时更新。
3. 后台健康探测仍继续使用 `last_probe_at`。

需要新增一条 goose migration 为现网补齐该字段。

## 后台管理调整

账号级手动绑定入口应从后台移除，范围包括：

1. 删除 `POST /api/admin/accounts/:id/bind-proxy`
2. 删除 `DELETE /api/admin/accounts/:id/bind-proxy`
3. 删除前端账号列表中的手动绑定表单与对话框
4. 删除新增账号表单中的“立即绑定代理”项

以下入口继续保留：

1. 导入表单中的 `default_proxy_id`
2. 与导入流程相关的代理列表拉取

## 验证要点

需要覆盖以下行为：

1. 临时代理按 `health_score DESC`、`last_used_at ASC`、`id ASC` 选择。
2. 所有健康代理忙碌时，`chat` 与 `image` 请求回退为直连。
3. `Lease.Release` 会释放代理锁与账号锁。
4. 仅 `chat` 与 `image` 申请临时代理，其他链路继续读持久绑定。
5. 业务失败不会扣减代理健康分。
6. 代理层失败每次请求最多扣减一次。
7. 后台账号页面已移除手动绑定入口，但导入默认代理仍然存在。
