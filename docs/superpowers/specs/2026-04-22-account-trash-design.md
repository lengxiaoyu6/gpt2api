# 账号池已删除列表设计文档

## 目标

在管理员账号池页面中增加“已删除”列表，承接当前软删除账号，并提供恢复与彻底删除能力。

当前账号池删除行为已经是软删除，缺少已删除视图与后续处理入口，导致管理员无法在界面中回收误删账号，也无法清理已确认废弃的账号数据。本次改动围绕现有软删除机制补齐回收区。

## 范围

本次改动包含以下内容：

1. 后端增加已删除账号分页列表接口。
2. 后端增加恢复接口，将 `deleted_at` 置空。
3. 后端增加彻底删除接口，物理删除软删除账号，并同步清理强关联数据。
4. 前端账号池页面增加“已删除”标签页，支持检索、恢复、彻底删除。
5. 前端删除提示文案调整为软删除语义。
6. 增加静态源码断言测试，覆盖接口符号、路由与页面入口。

本次改动保持以下内容不变：

1. 现有单条删除与批量删除仍然执行软删除。
2. `usage_logs`、`image_tasks` 等历史业务记录继续保留 `account_id` 引用，不参与清理。
3. 菜单、权限点、数据库迁移文件与现有调度逻辑维持现状。

## 数据边界

彻底删除的清理边界固定为以下三类强关联数据：

1. `oai_account_cookies`
2. `account_proxy_bindings`
3. `account_quota_snapshots`

账号主表 `oai_accounts` 仅允许对已经软删除的记录执行物理删除。活跃账号调用彻底删除接口时应返回未找到或无效状态，避免误删。

以下数据继续保留：

1. `usage_logs`
2. `image_tasks`
3. 其他仅通过 `account_id` 做历史追踪的业务表

这样可以保留审计与使用历史，同时去除账号的敏感配置与运行态附属数据。

## 接口设计

### 已删除列表

新增 `GET /api/admin/accounts/deleted`。

查询参数与现有账号列表保持一致：

1. `page`
2. `page_size`
3. `status`
4. `keyword`

筛选条件改为 `deleted_at IS NOT NULL`，排序按 `deleted_at DESC, id DESC` 返回，方便优先处理最近删除的账号。

### 恢复账号

新增 `POST /api/admin/accounts/:id/restore`。

恢复逻辑仅对 `deleted_at IS NOT NULL` 的账号生效，成功后返回恢复后的账号对象，使前端可以立即回到正常列表展示。

### 彻底删除账号

新增 `DELETE /api/admin/accounts/:id/purge`。

执行时使用事务，顺序为：

1. 先删除软删除账号主记录。
2. 再删除 cookies、代理绑定、额度快照。
3. 事务提交。

采用事务的目的是保证“账号主记录与强关联数据”同时成功或同时回滚。

## 页面行为

账号池页面增加两个标签页：

1. `账号列表`
2. `已删除`

`账号列表` 继续保留现有导入、刷新、探测、批量删除与编辑能力。

`已删除` 页面提供独立分页列表与筛选栏，字段至少包含邮箱、类型、状态、删除时间、备注、操作。操作列提供：

1. `恢复`
2. `彻底删除`

现有删除提示文案调整为软删除语义：删除后进入已删除列表，可恢复或彻底删除。批量删除提示同步调整。

## 错误处理

恢复或彻底删除目标不存在时返回 404。

彻底删除执行失败时保持事务回滚，避免出现主记录已经删除但 cookies 或绑定残留的半完成状态。前端统一显示失败提示，不在界面端推断失败原因。

## 测试与验证

本次继续使用现有 `node:test` 静态源码断言模式，覆盖以下方面：

1. `internal/account/dao.go` 存在已删除列表、恢复、彻底删除实现，并包含关联清理语句。
2. `internal/server/router.go` 挂载已删除列表、恢复、彻底删除路由。
3. `web/src/api/accounts.ts` 暴露已删除列表、恢复、彻底删除调用与 `deleted_at` 字段。
4. `web/src/views/admin/Accounts.vue` 包含“已删除”入口、“恢复”“彻底删除”操作，以及软删除提示文案。

完成修改后执行三组验证：

1. `cd web && node --test tests/account-trash.node.test.mjs`
2. `go test ./...`
3. `cd web && npm run build`
