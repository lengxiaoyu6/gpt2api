# 历史任务多图拆分展示设计

## 目标

将历史任务页中单条多图生成任务的展示粒度从“任务”调整为“图片”，让同一次任务产出的每一张结果图都在历史列表中拥有独立卡片，同时继续保留任务级预览与任务级删除语义。

## 现状

当前 `web/src/views/personal/HistoryTasks.vue` 直接以 `imageTasks` 为循环数据源，每条 `ImageTask` 只渲染一张卡片，缩略图固定使用 `t.image_urls[0]`，预览入口固定调用 `openPreview(t.image_urls, 0)`。

这导致当 `n > 1` 或 `image_urls.length > 1` 时，历史任务页只能看到第一张图，其余结果图仅能在放大预览后切换，列表本身无法明确展示多图任务的每一张结果。

## 目标行为

### 列表展示

页面继续通过 `/api/me/images/tasks` 获取任务列表，但前端在渲染前将每个任务拆成按图片粒度的展示项。

如果同一任务返回 4 张图，历史列表中连续渲染 4 张卡片。每张卡片展示当前图片自身的缩略图，并沿用原任务的提示词、状态、尺寸、时间与积分信息。

每张卡片额外展示同任务位置标识，格式为 `第 1 张，共 4 张`。当任务只有 1 张图时，也沿用这一结构，保证视觉规则一致。

### 预览交互

点击任意一张拆分后的卡片时，仍然打开该任务的整组图片预览器。

预览器的 `url-list` 继续使用当前任务完整的 `image_urls`，但 `initial-index` 应指向当前卡片对应的图片序号。这样从第 3 张卡片进入预览时，预览器默认打开第 3 张图。

### 删除交互

删除仍然按 `task_id` 生效。点击任意一张拆分卡片上的删除按钮，成功后该任务所属的全部拆分卡片一起从列表移除。

### 失败与空图任务

如果某条任务没有成功图片，即 `image_urls` 为空，则仍保留一张占位卡片，用于承载状态、错误信息和删除入口。这类占位项的任务位置信息按 `第 1 张，共 0 张` 不够自然，因此展示为 `结果图缺失`。

## 实现方式

### 页面数据结构

在 `HistoryTasks.vue` 中保留原始的 `imageTasks` 作为接口返回值缓存。

新增一个按图片展开的派生列表，例如 `flattenedImageTasks`，元素结构包含：

- `task`: 原始 `ImageTask`
- `image_url`: 当前卡片对应图片地址
- `image_index`: 当前图片序号，从 `0` 开始
- `image_total`: 当前任务图片总数
- `image_key`: 用于列表 `key` 的稳定标识
- `position_label`: 当前卡片的同任务位置文案

当 `image_urls.length > 0` 时，按每个 URL 展开一条记录。
当 `image_urls.length === 0` 时，生成一条占位记录，`image_url` 为空，`image_index` 为 `0`，`image_total` 为 `0`，`position_label` 为 `结果图缺失`。

### 模板调整

模板中的 `v-for` 从 `imageTasks` 改为 `flattenedImageTasks`。

卡片展示内容调整为：

- 缩略图显示 `item.image_url`
- 点击预览调用 `openPreview(item.task.image_urls, item.image_index)`
- 删除调用 `onDeleteTask(item.task)`
- 状态、提示词、时间、积分继续读取 `item.task`
- `n=` 文案替换为位置标识 `item.position_label`

### 样式调整

位置标识与尺寸信息保留在元信息区域，维持现有卡片高度与网格布局，避免本次需求扩展为完整的视觉重构。

## 测试范围

修改 `web/tests/history-tasks.node.test.mjs`，增加或调整以下断言：

- 页面存在按图片展开的派生列表逻辑
- 模板循环使用派生列表而非原始 `imageTasks`
- 预览点击传入当前图片序号
- 页面存在 `第 {{ ... }} 张，共 {{ ... }} 张` 或等价的位置标识逻辑
- 删除按钮绑定改为基于派生项中的原始任务对象

## 非目标

本次调整不修改后端接口结构，不新增新的筛选、分组、排序或视图切换能力，也不改变任务分页规则。
