# 图片文件本地保存与后台管理设计

## 目标

将图片任务的原图与缩略图写入本地文件系统，使历史任务页优先依赖本地文件展示，账号失效后仍可查看已保存图片。同时增加管理员图片文件页面，提供原图列表、缩略图列表、批量删除能力，以及当前分区容量与目录占用展示。

本次设计覆盖以下范围：

1. 图片任务成功后的原图保存与缩略图生成
2. 历史任务接口返回本地原图地址与缩略图地址
3. 本地图片访问接口
4. 管理员图片文件页面与后台接口
5. 文件删除后的前台展示语义
6. 磁盘容量与目录占用统计

## 当前实现

`internal/image/runner.go` 在图片任务成功后只将 `file_ids`、`result_urls`、`conversation_id` 写入 `image_tasks`，当前没有本地图片保存步骤。

`internal/gateway/images_proxy.go` 的 `/p/img/:task_id/:idx` 每次按 `task_id` 查任务，再依赖账号凭据向上游换取下载地址并拉取原图。账号失效、会话失效或上游下载地址换取失败时，历史图片无法显示。

`web/src/views/personal/HistoryTasks.vue` 当前只消费 `image_urls`，列表卡片与预览共用同一组地址。

后台当前已有 `数据备份` 页面与对应接口模式，可作为本次管理员文件页面的实现参考，但现有系统中还没有图片文件管理功能，也没有目录占用统计实现。

## 方案比较

### 方案一

在图片任务成功阶段立即抓取原图、写入本地、生成缩略图；数据库继续只保存任务本身信息，本地文件名按固定规则推导，历史页与管理员页面都通过文件系统读取实际状态。

优点：

1. 账号失效后，历史图片仍然可用
2. 文件命名稳定，管理员手工排查容易
3. 数据库改动小，只需增加接口视图字段即可
4. 原图与缩略图可以分别删除，历史页语义清晰

代价：

1. 任务成功时间会多出一次原图下载与一次缩略图生成
2. 需要补充文件系统扫描、容量统计与文件读取服务
3. 旧任务若本地没有文件，将显示“已过期”

### 方案二

在方案一基础上，为 `image_tasks` 增加本地文件元数据字段，例如原图文件名数组、缩略图文件名数组、文件状态数组。历史页与后台页面优先读取数据库中的文件信息。

优点：

1. 历史页组装时减少文件系统探测次数
2. 管理后台列表可以更快分页
3. 文件状态表达更细，可以记录部分成功、部分缺失

代价：

1. 需要新增迁移、DAO、模型字段与数据一致性维护
2. 文件系统状态与数据库状态容易出现偏差，需要额外校验与修复逻辑
3. 删除文件后还要同步更新数据库，维护成本更高

### 方案三

维持当前任务成功流程，直到历史页或预览页首次访问图片时再尝试写入本地文件。

优点：

1. 生图成功返回更快
2. 对从未访问过的任务节省磁盘空间

代价：

1. 首次访问仍依赖账号状态，无法解决账号失效问题
2. 同一张图的首次访问体验仍然较慢
3. 逻辑分散在历史页展示阶段，状态更难判断

## 采用方案

采用方案一。

原因有两点。其一，当前需求的核心是“账号失效后仍可显示图片”，因此图片保存动作必须发生在账号与会话仍然有效的任务成功阶段。其二，当前项目中的 `image_tasks` 已经具备 `task_id`、`file_ids`、`n` 等足以推导文件名的信息，继续使用固定命名规则配合文件系统扫描，整体改动更集中，维护负担更小。

## 详细设计

### 目录与命名

新增两个固定目录：

1. `storage/images/original/`
2. `storage/images/thumb/`

文件命名规则：

1. 原图：`<task_id>_<idx>.<ext>`
2. 缩略图：`tmp_<task_id>_<idx>.jpg`

其中 `idx` 从 `0` 开始，与 `image_urls` 的下标保持一致。

原图扩展名由真实图片类型决定，至少支持 `png`、`jpg`、`webp`。缩略图统一转成 JPEG，便于控制体积与后台列表展示。

### 保存时机与任务成功语义

图片任务在上游返回成功引用后，增加一个“图片归档”步骤：

1. 使用当前任务已有的下载信息抓取每一张原图字节
2. 将原图写入 `storage/images/original/`
3. 由原图生成缩略图并写入 `storage/images/thumb/`
4. 全部完成后，任务才写入 `success`

图片归档失败时，任务记为失败，错误码沿用 `download_failed` 或新增 `archive_failed`。此时需要删除本次任务已写入的临时文件，避免出现半套文件。

保存过程采用临时文件加原子重命名的方式，例如先写入 `.tmp` 文件，全部成功后再改成正式文件名。这样可以保证管理员列表与历史页只看到完整文件。

### 缩略图规格

缩略图统一使用 JPEG。

推荐规格：长边 480 像素，质量 82，按原始比例缩放，不裁切、不加水印。

这样可以兼顾列表首屏速度、磁盘占用与后台管理可读性。历史任务卡片优先使用缩略图，预览仍然使用原图。

### 本地文件服务

新增图片文件服务，职责包括：

1. 保存任务原图
2. 生成并保存缩略图
3. 按 `task_id + idx` 解析原图文件与缩略图文件
4. 删除原图文件或缩略图文件
5. 列出目录文件
6. 统计目录占用与所在分区容量

推荐放在新包中，例如 `internal/imagestore`。

该服务维护以下核心方法：

1. `SaveTaskImages`
2. `FindOriginal`
3. `FindThumb`
4. `BuildOriginalURL`
5. `BuildThumbURL`
6. `ListFiles`
7. `DeleteFiles`
8. `DiskStats`

### 图片访问地址

原图继续沿用现有地址：`/p/img/:task_id/:idx`

这样可以保留以下现有能力：

1. 已有前端与开发代理配置无需调整主体规则
2. `upscale` 仍然可以继续在原图读取后执行
3. 外部客户端收到的图片地址形式保持稳定

同时新增缩略图地址：`/p/thumb/:task_id/:idx`

两类地址都继续使用带过期时间与签名的方式，避免任意枚举。签名计算需要把资源类型纳入签名内容，防止原图签名被拿去读取缩略图，或反向混用。

读取逻辑统一调整为“优先本地文件”：

1. 校验签名
2. 按 `task_id + idx` 查找本地原图或缩略图
3. 原图命中时，按现有 `upscale` 规则决定是否进行放大
4. 文件缺失时返回 `404`

这里不再从本地缺失回退到上游抓取。这样管理员删除文件后的语义才稳定，历史页才能明确显示“已过期”。

### 历史任务接口

`GET /api/me/images/tasks` 与 `GET /api/me/images/tasks/:id` 在现有 `image_urls` 基础上新增 `thumb_urls` 字段。

字段语义：

1. `image_urls` 表示原图访问地址
2. `thumb_urls` 表示缩略图访问地址
3. 两个数组下标一一对应

组装规则：

1. 如果原图存在，`image_urls[idx]` 返回 `/p/img/...`
2. 如果缩略图存在，`thumb_urls[idx]` 返回 `/p/thumb/...`
3. 如果原图存在而缩略图缺失，`thumb_urls[idx]` 回退为原图地址，保证历史页仍能正常显示
4. 如果原图缺失，则 `image_urls[idx]` 与 `thumb_urls[idx]` 都返回空字符串

这样可以形成稳定语义：

1. 原图缺失：该图片显示“已过期”
2. 缩略图缺失：历史页仍可显示，但会使用原图作为卡片图
3. 失败任务且本来没有图片：继续显示当前失败状态

前端 `web/src/views/personal/HistoryTasks.vue` 调整为：

1. 卡片缩略图优先使用 `thumb_urls[idx]`
2. 预览使用 `image_urls`
3. `task.status === 'success'` 且当前图片地址为空时，占位文案显示“已过期”
4. 其他空图场景仍按现有状态显示

### 管理员后台页面

新增后台菜单项与页面，例如：

1. 菜单：`图片文件`
2. 路由：`/admin/image-files`
3. 页面文件：`web/src/views/admin/ImageFiles.vue`

页面结构分三块：

#### 顶部容量概览

展示当前图片目录所在分区信息：

1. 总容量
2. 已用容量
3. 剩余容量
4. 使用率
5. `storage/images/original/` 占用
6. `storage/images/thumb/` 占用

#### 原图文件列表

展示字段：

1. 文件名
2. 对应任务 ID
3. 图片序号
4. 文件大小
5. 修改时间

支持多选，支持批量删除。

#### 缩略图文件列表

展示字段与原图列表一致，也支持多选批量删除。

交互参考 `Backup.vue` 的表格模式，删除只需确认弹窗，无需管理员密码二次输入。原因是该功能只影响图片文件，不涉及数据库恢复、账号密钥或积分变更。

### 管理员后台接口

推荐新增一组后台接口：

1. `GET /api/admin/system/image-files/stats`
2. `GET /api/admin/system/image-files/original`
3. `GET /api/admin/system/image-files/thumb`
4. `POST /api/admin/system/image-files/original/delete`
5. `POST /api/admin/system/image-files/thumb/delete`

列表接口支持 `limit` 与 `offset`，返回总数与当前页数据。

单条文件记录推荐包含：

1. `name`
2. `task_id`
3. `idx`
4. `size_bytes`
5. `updated_at`

批量删除请求体：

```json
{
  "names": ["img_xxx_0.png", "img_xxx_1.png"]
}
```

删除动作只接受文件名，不接受任意相对地址，服务端需再次校验文件名是否属于当前目录，避免目录穿越。

### 权限与菜单

推荐新增权限 `system:image_file`。

原因是该功能与 `system:backup` 的危害等级、使用频率、管理对象都不同，分开控制更清晰。对应改动包括：

1. `internal/rbac/permission.go` 增加权限常量，并赋给 `admin`
2. `internal/rbac/menu.go` 增加菜单项
3. `web/src/router/index.ts` 增加管理页面路由
4. `internal/server/router.go` 增加后台接口分组

如果当前版本希望减少权限面变化，也可以临时复用 `system:backup`。本次设计仍推荐单独权限。

### 磁盘容量统计

容量统计分成两类：

#### 分区容量

以 `storage/images/original/` 所在分区为统计基准，使用 `statfs` 读取：

1. `total_bytes`
2. `used_bytes`
3. `free_bytes`
4. `used_percent`

#### 目录占用

递归统计两个目录下所有文件大小总和：

1. `original_dir_bytes`
2. `thumb_dir_bytes`
3. `original_file_count`
4. `thumb_file_count`

考虑到图片目录规模可增长，目录占用统计可以在每次后台页面加载时重新计算；如果后续数据量很大，再增加短时缓存。

### 删除行为

管理员删除行为遵循以下规则：

1. 只删除磁盘文件
2. `image_tasks` 记录保留
3. 原图与缩略图不做联动删除
4. 删除缩略图后，如果原图仍在，历史页使用原图作为卡片图
5. 删除原图后，该图片在历史页显示“已过期”
6. 删除整个任务的全部原图后，该任务相关图片卡片全部进入“已过期”状态

### 存量任务处理

该设计保证功能上线后的新任务会拥有本地原图与缩略图。

历史已有任务如果本地文件不存在，将按“已过期”显示。当前设计不包含历史任务自动补齐程序，也不在请求时临时回退抓取旧图。

这样可以保持语义一致，也能避免本次实现范围扩大。

### 错误处理

需要重点处理以下情况：

1. 原图下载失败：任务记失败，清理临时文件
2. 缩略图生成失败：任务记失败，清理临时文件
3. 单图任务部分文件写入成功、部分失败：整任务按失败处理
4. 管理员删除时文件已不存在：按幂等成功处理，并在返回中给出已删除数量
5. 容量统计失败：后台页面显示“读取失败”，文件列表仍可继续使用
6. 文件名解析失败：后台列表跳过异常文件，同时记录日志

### 测试范围

#### 后端单元测试

1. 文件命名与解析
2. 原图扩展名识别
3. 缩略图生成尺寸与格式
4. 原图与缩略图查找逻辑
5. 原图缺失、缩略图缺失时的历史任务地址组装
6. 批量删除的目录校验与幂等行为
7. 分区容量与目录占用统计
8. 原子保存失败后的清理行为

#### 接口测试

1. 图片任务成功后，本地生成原图与缩略图
2. 历史任务接口返回 `image_urls` 与 `thumb_urls`
3. `/p/img/...` 可读取本地原图并保留 `upscale` 能力
4. `/p/thumb/...` 返回缩略图
5. 管理员页面列表与批量删除接口行为正确

#### 前端验证

1. 历史任务卡片优先显示缩略图
2. 原图被删后卡片显示“已过期”
3. 缩略图被删后卡片仍能显示
4. 管理页顶部容量数据与表格刷新正常
5. 原图列表与缩略图列表都支持多选删除

## 范围外事项

本次设计不包含以下内容：

1. 图片文件自动压缩清理策略
2. 历史任务自动补齐旧数据
3. 对象存储接入
4. 管理员文件下载功能
5. 图片文件与任务记录的一对一数据库元数据表
